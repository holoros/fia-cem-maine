## The unit bug (the lesson of this session)

`R/01_data_prep.R` builds `carbon_ag = sum(TPA_UNADJ * CARBON_AG)` where `CARBON_AG` is the FIA tree level carbon column in **pounds per tree**. The aggregated `carbon_ag` (and downstream `proj_carbon` in `per_plot_projections.rds` and `mean_carbon_mean` in `table_inventory_summary.csv`) is therefore in **lb/ac**, not kg/ac.

My hindcast and validation template scripts used `1e-9` for the kg to MMT conversion, treating the values as kg. The correct factor for lb is `LB_TO_MMT = 4.53592e-10`. Ratio of the two is exactly 2.20, which is exactly the spurious over prediction factor I was seeing across all multistate hindcasts.

The fix landed in commit `259b481`:
- `scripts/hindcast_multistate.R`: `LB_TO_MMT` applied everywhere, `proj_carbon_lbac` variable name
- `scripts/validate_template.R`: `total_carbon_tgc` formula uses `4.5359e-10`
- `STATE_PROFILES` total_carbon_tgc bounds rebased on FIA full panel observed (WA 500-800, MN 180-320, GA 330-500, ME 300-420 TgC)

The retracted findings (the +65 to +145 percent hindcast bias, the WA carbon overshoot, the MN structural carbon issue) are now documented as analysis artifacts in `docs/UNIT_BUG_FINDING_20260515.md`. The post fix corrected results are in `docs/VALIDATION_FINAL_20260515.md`.

The episode is itself publishable methodology for the manuscript: validating CEM projections requires careful unit tracking through the pipeline, and the diagnostic process of catching this bug is worth a few sentences in the methods or appendix.

## Layer 2 10 sim smoke + Layer 3 patch discovered (landed mid session)

SLURM job 9608591 produced cycle 1 BAU harvest rate of **0.8359 with tight CI [0.8342, 0.8367] across 10 sims** — essentially deterministic, NOT variance. This triggered a focused audit of the Wear 2025 logit code path that uncovered the **Layer 3 patch**: the same `tpa_live` double counting bug existed at `R/03_harvest_choice.R` line 108 in `compute_ending_value()` that Layer 2 fixed at line 409.

```r
# Pre Layer 3:
EV = coalesce(T2_volcfnet, volcfnet) * (price_avg) * tpa_live
# Post Layer 3:
EV = coalesce(T2_volcfnet, volcfnet) * (price_avg)
```

The two `tpa_live` bugs had been compensating by coincidence in the pre Layer 2 code: line 409 inflated `vol_removed_total` and hence `REV_harvest`, line 108 inflated `EV`. Both feed into `dVAL = abs(REV_harvest + delta * (EV_harvest - EV_noharvest))`. With both inflated, the logit happened to produce a roughly realistic 10 percent harvest rate by accident. Layer 2 alone broke the compensation; Layer 3 restores correct per acre dynamics throughout the harvest economic overlay.

**Layer 3 patch applied this session.** Live `R/03_harvest_choice.R` updated, backup preserved at `R/03_harvest_choice.R.preupdate.20260515_layer3`. Layer 3 verification smoke completed as SLURM job 9669294.

**Layer 3 result: patch fixes dead code.** Cycle 1 BAU harvest rate landed at 0.836 with CI [0.834, 0.838], essentially identical to the Layer 2 only result. A subsequent grep audit confirmed `compute_ending_value()` is defined at R/03 line 99 but never called anywhere in the codebase. The Layer 3 patch corrects a real unit error in dead code.

**Layer 4 patch applied later this session.** Reading `predict_harvest_probability()` revealed the active dVAL site at R/03 line 257 (`dVAL = REV_harvest` as a Wear 2025 proxy). REV_harvest is computed by multiplying per acre cuft volumes against price defaults in R/00_config.R that are documented as $/MBF (sawtimber) and $/cord (pulpwood) — a 200x and 80x unit mismatch respectively. Predicted effect: REV_harvest drops from ~$400,000/ac to ~$1,200/ac, logit term drops from saturation to ~2.0, P(harvest) drops from 0.84 to predicted ~0.56. The Layer 4 patch (`R/03_harvest_choice.R` lines 74-91) adds `MBF_per_CUFT = 1/200` and `CORD_per_CUFT = 1/80` conversions. Backup at `R/03_harvest_choice.R.preupdate.20260516_layer4`. Layer 4 verification smoke running as SLURM job 9671919.

**Layer 5 likely still needed.** Even after Layer 4, the back of envelope prediction (P~0.56) is still high vs observed Maine ~0.10. The remaining issue is that `dVAL = REV_harvest` at line 257 is a "simplified using revenue as proxy" placeholder per the code comment. The proper Wear 2025 formula is `dVAL = abs(REV + delta * (EV_h - EV_nh))` which involves canceling terms (post harvest EV is mostly $0 regen, no harvest EV is discounted standing timber) producing a smaller actual differential. Implementing the proper differential is Layer 5 pending; estimated 1 to 2 hour analytical task once Layer 4 result confirms the price unit fix.
